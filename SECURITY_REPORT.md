## Summary
M01 implements a greenfield Python CLI tool skeleton with config loading, snapshot data structures, and atomic file I/O. There are no network calls, no authentication surfaces, no cryptographic operations, and no user-controlled strings passed to shell commands. The attack surface is narrow and local. All findings are LOW severity — none are blocking for a local developer CLI tool. The most notable pattern to watch for in future milestones is the unvalidated environment variable paths (`SDI_CONFIG_PATH`, `SDI_SNAPSHOT_DIR`) which allow reading/writing outside the project directory for any caller who controls the environment.

## Findings
- [LOW] [category:A05] [src/sdi/config.py:284-285] fixable:yes — `SDI_CONFIG_PATH` is accepted without path constraint validation. An actor who controls the environment can point this at any file on the filesystem (e.g. `/etc/shadow`). The file is opened in binary mode by `tomllib`, so a parse error exits cleanly, but a syntactically-valid TOML file anywhere on disk would be accepted as config. For a developer CLI this is low risk, but a path-within-project-or-home check would harden it.
- [LOW] [category:A05] [src/sdi/config.py:164-165] fixable:yes — `SDI_SNAPSHOT_DIR` is accepted without path constraint validation. This value is subsequently passed to `write_snapshot → write_atomic → tempfile.mkstemp(dir=...)`, which will create directories and write files at the attacker-supplied path (given write permission). A check that the resolved path is under the project root or a configurable base would prevent out-of-tree writes.
- [LOW] [category:A05] [src/sdi/config.py:157-158] fixable:yes — `SDI_LOG_LEVEL` value is stored in `CoreConfig.log_level` without validation against Python's allowed log level names (`DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`). In the current milestone this is inert since no logging is configured, but when logging is wired up in later milestones an arbitrary string here will silently fall back to the default level (or raise `ValueError` in `logging.setLevel`). Add an allowlist check at load time with exit code 2 on invalid value.
- [LOW] [category:A05] [src/sdi/config.py:184-188] fixable:yes — In `_build_overrides`, a `ValueError` from `date.fromisoformat(entry["expires"])` causes the override to be silently skipped via `continue`. A user who typos the date format (e.g. `"2026/09/30"` instead of `"2026-09-30"`) will have their threshold override silently not applied — they believe the guardrail is relaxed but the default (stricter) threshold applies. `_validate_overrides` only checks key presence, not date parseability. The fix is to move date-format validation into `_validate_overrides` so a malformed date is a config error (exit code 2) rather than a silent no-op.
